The DEC (Detección de Enfermedades en Cafetales) API is an Express.js REST API built with Node.js that exposes structured JSON endpoints for managing users, disease detections, pathologies, treatments, and administrative operations. The server listens on port **8089** by default (configurable via the `PORT` environment variable). All request and response bodies are JSON. Every protected endpoint requires a valid JWT passed in the `Authorization: Bearer` header.

## Base URL

For local development the server binds to all interfaces on port 8089:

```
http://localhost:8089
```

In production, replace `localhost` with your server's hostname or IP address. The port may be overridden by the `PORT` environment variable at startup.

## Authentication

All protected routes require a signed JWT in the `Authorization` header:

```
Authorization: Bearer <token>
```

Tokens are issued with a **30-day expiry** by `jwt.sign` in the login and OAuth controllers. Admin-only routes additionally check the `role` claim inside the token — requests from non-admin users are rejected with `403 Forbidden`.

See the [Authentication reference](/api/authentication) for full details on obtaining and using tokens.

## Full Endpoint Index

The table below lists every route registered in the DEC backend. The **Auth** column indicates whether a valid JWT (`🔒`) and/or admin role (`🛡️`) is required.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `POST` | `/api/users/register` | Public | Create a new account. Sends a 6-digit verification code to the provided email address. |
| `POST` | `/api/users/verify-code` | Public | Confirm account email with the 6-digit code received after registration. |
| `POST` | `/api/users/login` | Public | Authenticate with email and password. Returns a JWT and basic user object. |
| `POST` | `/api/users/auth/google` | Public | Sign in or register with a Google ID token passed in the `Authorization: Bearer` header. |
| `POST` | `/api/users/auth/facebook` | Public | Sign in or register with a Facebook access token passed in the `Authorization: Bearer` header. |
| `POST` | `/api/users/auth/facebook-deletion` | Public | Facebook-required data-deletion callback. Validates the signed request and removes the user from the database. |
| `GET` | `/api/users/me` | 🔒 | Return the full profile of the currently authenticated user (excludes password and internal verification fields). |
| `PUT` | `/api/users/edit-profile` | 🔒 | Update the authenticated user's `name` and/or `phone` fields. |
| `POST` | `/api/users/change-password` | 🔒 | Change the current password by supplying the existing password and a new one. |
| `POST` | `/api/users/upload-profile-picture` | 🔒 | Upload a profile picture (`multipart/form-data`, field `image`). Stores the image on Cloudinary and returns the new URL. |

## Error Responses

All error responses follow the same JSON structure:

```json
{
  "message": "Human-readable description of the error."
}
```

| Status | Meaning in DEC |
|--------|----------------|
| `400 Bad Request` | A required field is missing or fails validation (e.g. invalid email format, password too short, duplicate registration). |
| `401 Unauthorized` | No `Authorization` header was provided, the token is absent, or the supplied credentials (email/password) are incorrect. The response body contains `"Acceso denegado. No hay token."` or a credential error message. |
| `403 Forbidden` | The token is present but invalid or expired, **or** the authenticated user does not have the `admin` role required by the endpoint. |
| `404 Not Found` | The requested resource (user, detection, pathology, etc.) does not exist in the database. |
| `500 Internal Server Error` | An unexpected server-side error occurred. The response may include an `error` field with the original error message for debugging. |
