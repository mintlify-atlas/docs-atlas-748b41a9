The DEC mobile app provides coffee farmers and agronomists with a complete plant health toolkit ‚Äî from AI-powered disease detection using an on-device YOLOv11 model to offline-first local storage, treatment tracking, scheduled reminders, and synchronized history. All functionality is available on iOS and Android; the web build delivers history browsing and an admin dashboard but omits on-device inference.

## Authentication Screens

<CardGroup cols={2}>
  <Card title="Login" icon="right-to-bracket">
    Sign in with **email and password**, **Google Sign-In** (via `@react-native-google-signin/google-signin`), or **Facebook** (via `react-native-fbsdk-next`). Tokens are stored securely with `expo-secure-store` on mobile and `sessionStorage` on web. Expired tokens trigger an automatic logout and redirect to this screen.
  </Card>
  <Card title="Register" icon="user-plus">
    Two-step registration flow: the user submits their name, email, and password (`POST /api/users/register`), then enters a verification code sent to their inbox (`POST /api/users/verify-code`). The `VerifyCodeScreen` handles the second step.
  </Card>
  <Card title="Forgot Password" icon="key">
    Email-based recovery split across two screens. **ForgotPassword** sends a one-time code (`POST /api/recover/req-code`). **ResetPassword** accepts the code and a new password (`POST /api/recover/change-pass`).
  </Card>
  <Card title="Onboarding" icon="star">
    Displayed on first launch via the `OnboardingScreen` component (`src/components/OnboardingScreen.jsx`), which uses `react-native-onboarding-swiper`. Once completed, a flag is written to `AsyncStorage` so the swiper is not shown again. The user is then directed to the Login screen.
  </Card>
</CardGroup>

## Core Detection Flow

<Steps>
  <Step title="Main screen ‚Äî tap the camera button">
    After login, the user lands on **MainApp** (the tab navigator). Tapping the camera shortcut navigates to the `Camera` stack screen.
  </Step>
  <Step title="Camera screen ‚Äî frame and capture a leaf">
    `CameraScreen` opens the rear-facing camera via `expo-camera`. A 220 √ó 220 pt dashed green reticle guides the user to center a single leaf in the frame. A loading overlay (`"Cargando inteligencia artificial..."`) is shown while the TFLite model initializes before the capture button becomes active.
  </Step>
  <Step title="Preprocessing ‚Äî image resized and normalized">
    On capture, `expo-image-manipulator` crops the image to the reticle area and then resizes it to 640 √ó 640 pixels. An invisible `GLView` (Expo2DContext) rasterizes the asset and extracts a normalized `Float32Array` with pixel values in `[0, 1]`. See the [AI Detection](/mobile/ai-detection) page for the full pipeline.
  </Step>
  <Step title="Results screen ‚Äî disease, confidence, and treatment">
    `navigation.navigate('Result', { data })` passes the detection payload to the **Result** screen, which displays:
    - **Disease name** (e.g., `Roya (Leaf Rust)`, `Cercospora`, `Minador de la hoja`, `Ara√±a roja`, or `Planta Sana`)
    - **Scientific name** (e.g., *Hemileia vastatrix*)
    - **Confidence percentage** returned by the model
    - **Brief description** of the pathology
  </Step>
  <Step title="Saved to SQLite history automatically">
    The detection is persisted locally via `saveDetectionLocal()` from `dbService.js` into the `detections` table (`dec_app.db`). When the device is online, `syncService.js` uploads pending records to `POST /api/detections/save` and marks them as synced.
  </Step>
</Steps>

## History & Profile Screens

### History

**HistoryScreen** (`src/screens/mobile/HistoryScreen.jsx`) displays all past analyses in a scrollable card list. Each card shows the captured leaf image, disease name, date (formatted with `Intl.DateTimeFormat` in `es-CO` locale), and confidence badge.

- **Online mode**: fetches from `GET /api/detections/history?page=1&limit=100` and triggers a background `syncServerToLocal()` to keep SQLite current.
- **Offline mode**: reads directly from the `remote_detections` SQLite table via `getAllRemoteDetections()`.
- Pull-to-refresh re-syncs with the server when connected.
- Tapping a card navigates to **DetectionDetail**.

### Detection Detail

**DetectionDetailScreen** (`src/screens/mobile/DetectionDetailScreen.jsx`) shows the full record for a single analysis:

| Section | Details |
|---------|---------|
| Hero image | Full-width leaf photo with a colored badge (warning yellow for disease, success green for healthy) |
| Confidence bar | Animated progress bar showing inference accuracy as a percentage |
| Treatment log | Links to the associated `TreatmentLogScreen` entry, or a button to create one |
| Suggested products | Modal listing recommended products (name, active ingredient, dose, price, supplier, link) pulled from `getPathologyWithRecommendations()` |
| Reminders | Schedule a `expo-notifications` push reminder for follow-up; stored in the local `alarms` SQLite table |
| Location | "Ver en mapa" button opens Google Maps at the detection's GPS coordinates (recorded from `expo-location`) |

### Profile

**Profile** displays the user's name, email, profile photo, total detections, and total treatment log entries. The counters come from the local `user_stats` SQLite table (synced on login). **EditProfile** allows updating the display name, phone number, and profile picture (uploaded to the backend via `multipart/form-data`).

## Treatment Logbook (Bit√°cora)

**TreatmentLogScreen** and **TreatmentFormScreen** provide a multi-product treatment journal linked to a specific detection. Each log entry stores:

- **Disease name** targeted by the treatment
- **General notes** about the crop condition
- One or more **products** (name, dose, application date, notes)

Logs are saved locally in the `treatment_logs` and `treatment_products` SQLite tables, then synced to `POST /api/treatments` (new) or `PUT /api/treatments/:id` (update) when online.

`MobileNavigator` uses a `createStackNavigator` with `headerShown: false` throughout. The active branch (Auth vs App Stack) is controlled by the `userToken` and `isGuest` values from `AuthContext`.

## Offline Support

<Note>
  All detections and the pathology catalogue are stored in **SQLite** (`dec_app.db`) using `expo-sqlite`. When connectivity is restored, `syncService.js` automatically re-syncs in four passes:

  1. `syncDetections()` ‚Äî uploads locally saved detections to `POST /api/detections/save`
  2. `syncServerToLocal()` ‚Äî downloads the full detection history from `GET /api/detections/history` and merges into `remote_detections`
  3. `syncLocalTreatments()` ‚Äî pushes local treatment logs to the backend
  4. `syncRemoteTreatments()` ‚Äî downloads server-side treatment logs into local SQLite

  `startAutoSync(token)` registers a `NetInfo` listener so synchronization begins automatically whenever an internet connection is detected, with exponential back-off retries (up to 3 attempts per item). The pathology catalogue (names, descriptions, treatments, recommended products) is also cached locally so the **Results** and **DetectionDetail** screens load instantly without a network request.
</Note>
