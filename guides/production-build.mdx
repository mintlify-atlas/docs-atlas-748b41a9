DEC supports three production targets — Android APK/AAB, iOS IPA, and a static web export. Each target has a distinct build pipeline. Backend deployment is a standard Node.js process that can be managed with PM2 or any cloud provider.

## Mobile Production Builds with EAS

DEC's mobile app is compiled using [Expo Application Services (EAS) Build](https://docs.expo.dev/build/introduction/). EAS runs the build in the cloud, so you do not need a local Mac to produce iOS builds.

<Steps>
  <Step title="Install EAS CLI">
    ```bash
    npm install -g eas-cli
    ```
    The `eas.json` file requires CLI version `>= 18.3.0`.
  </Step>
  <Step title="Log in to your Expo account">
    ```bash
    eas login
    ```
    Use the account that owns the EAS project with ID `03e2f248-6cd0-4b8c-a177-41660d5ee325` (set in `app.json` under `extra.eas.projectId`).
  </Step>
  <Step title="Configure EAS (first time only)">
    If `eas.json` does not exist yet, run:
    ```bash
    eas build:configure
    ```
    For DEC, `eas.json` is already committed to the repository — skip this step if the file is present.
  </Step>
  <Step title="Build for Android">
    ```bash
    cd frontend/DEC
    eas build --platform android
    ```
    This uses the `production` profile, which sets `android.buildType: "apk"` and `environment: "production"`.
  </Step>
  <Step title="Build for iOS">
    ```bash
    cd frontend/DEC
    eas build --platform ios
    ```
    The production profile does not specify an iOS `buildType`, so EAS defaults to an `.ipa` archive suitable for App Store submission. The bundle identifier is `com.decapp.app` (from `app.json`).
  </Step>
  <Step title="Download the build artifact">
    When the build completes, EAS CLI prints a direct download URL. You can also find all builds in the [Expo dashboard](https://expo.dev) under your project.
  </Step>
</Steps>

## EAS Build Profiles

The full `eas.json` configuration shipped with DEC:

```json
{
  "cli": {
    "version": ">= 18.3.0",
    "appVersionSource": "remote"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal"
    },
    "preview": {
      "android": {
        "buildType": "apk"
      }
    },
    "production": {
      "android": { "buildType": "apk" },
      "environment": "production"
    }
  },
  "submit": {
    "production": {}
  }
}
```

| Profile | Purpose | Android output |
|---|---|---|
| `development` | Internal distribution builds with dev client enabled. Used for testing with `expo-dev-client`. | Not specified (defaults to AAB) |
| `preview` | Ad-hoc APK for testers before a production release. | `.apk` |
| `production` | Store-ready release. `appVersionSource: "remote"` means the version is managed in the EAS dashboard. | `.apk` |

To target a specific profile explicitly:

```bash
eas build --platform android --profile preview
```

## Web Export

The DEC web build is a static export generated by Expo and then post-processed by a `copy-fonts.js` script that ensures icon fonts are available in the `dist/` output.

<Steps>
  <Step title="Run the web build script">
    From the `frontend/DEC` directory:

    ```bash
    npm run build:web
    ```

    This executes:
    ```bash
    expo export --platform web && node copy-fonts.js
    ```
  </Step>
  <Step title="Locate the output">
    The compiled static assets are written to the `dist/` directory. This folder contains an `index.html` entry point, chunked JavaScript bundles, and all static assets including fonts and images.
  </Step>
  <Step title="Deploy to a static host">
    The `dist/` directory can be served by any static hosting provider:

    ```bash
    # Vercel
    vercel --prod

    # Netlify
    netlify deploy --prod --dir dist

    # Nginx — copy dist/ to your web root
    cp -r dist/* /var/www/html/dec/
    ```

    No server-side rendering is required; the web client communicates directly with the DEC backend API.
  </Step>
</Steps>

## Backend Deployment

The DEC backend is a Node.js + Express application located in `backend/`. Its entry point is `server.js`.

### Set production environment variables

Create a `.env` file (or inject variables through your host's environment configuration):

```bash
# Database
MONGODB_URI=mongodb+srv://<user>:<password>@cluster.mongodb.net/dec

# Auth
JWT_SECRET=your_production_jwt_secret

# Cloudinary (file storage)
CLOUDINARY_CLOUD_NAME=your_cloud_name
CLOUDINARY_API_KEY=your_api_key
CLOUDINARY_API_SECRET=your_api_secret

# OAuth
GOOGLE_CLIENT_ID=your_google_client_id
FACEBOOK_APP_ID=your_facebook_app_id

# Email (Nodemailer / Resend)
SMTP_HOST=smtp.gmail.com
SMTP_USER=your_email@gmail.com
SMTP_PASS=your_app_password
RESEND_API_KEY=your_resend_api_key
FROM_EMAIL_ADMIN=admin@dec-app.online

# Server
PORT=3000
```

### Start with npm

```bash
cd backend
npm start
```

`npm start` runs `node server.js` as defined in `backend/package.json`.

### Recommended: use PM2

For a resilient production process that survives crashes and server restarts:

```bash
npm install -g pm2

# Start the backend
pm2 start server.js --name dec-backend

# Persist across reboots
pm2 save
pm2 startup

# Monitor logs
pm2 logs dec-backend
```

### Cloud provider PORT

Most cloud providers (Railway, Render, Heroku, etc.) inject a dynamic `PORT` environment variable. Make sure `server.js` reads it:

```js
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`DEC backend running on port ${PORT}`));
```

## Pre-Launch Environment Checklist

<AccordionGroup>
  <Accordion title="MongoDB Atlas">
    - Create a production cluster on [MongoDB Atlas](https://cloud.mongodb.com).
    - Add your backend server's IP (or `0.0.0.0/0` for dynamic IPs) to the **Network Access** allowlist.
    - Create a dedicated database user with read/write access — do not use the Atlas admin account.
    - Set `MONGODB_URI` to the Atlas connection string, including the database name: `mongodb+srv://...@cluster.mongodb.net/dec`.
  </Accordion>
  <Accordion title="Cloudinary">
    - Ensure `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, and `CLOUDINARY_API_SECRET` are production credentials, not the development or free-tier sandbox.
    - Verify upload presets allow the file sizes produced by `expo-camera` (default is JPEG from mobile cameras).
  </Accordion>
  <Accordion title="Email / SMTP">
    - Configure `SMTP_HOST`, `SMTP_USER`, and `SMTP_PASS` for the production email domain used in password recovery flows.
    - If using Resend, set `RESEND_API_KEY` and verify `FROM_EMAIL_ADMIN` matches a domain confirmed in your Resend account (the default is `admin@dec-app.online`).
  </Accordion>
  <Accordion title="OAuth redirect URIs">
    - In the Google Cloud Console, add the production backend domain to the list of **Authorized redirect URIs** for your OAuth 2.0 client.
    - In the Facebook Developer Portal, update the **Valid OAuth Redirect URIs** and **App Domains** to include your production domain.
    - The Facebook SDK is configured in `app.json` with `appID: "4404660439775314"` — ensure this App ID is live (not in development mode) before publishing to the stores.
  </Accordion>
</AccordionGroup>

<Warning>
  Before running `eas build`, update `EXPO_PUBLIC_API_URL` in the frontend `.env` file to point to your production backend URL:

  ```bash
  EXPO_PUBLIC_API_URL=https://api.your-production-domain.com
  ```

  This variable is baked into the JavaScript bundle at build time by Expo. If you build with a `localhost` or staging URL, all API calls from the production app will fail.
</Warning>
