DEC uses [Expo Notifications](https://docs.expo.dev/versions/latest/sdk/notifications/) to deliver push notifications to iOS and Android users. Device tokens are stored server-side and notifications can be created and broadcast from the admin dashboard, targeting users by role.

## Client-Side Setup with `notificationService.js`

The `notificationService.js` file configures how the app handles incoming notifications and provides the function that registers a device for push delivery.

### Foreground notification handler

Regardless of whether a notification arrives while the app is in the foreground or background, the handler is configured to always display an alert, play a sound, and update the badge count:

```js
import * as Notifications from 'expo-notifications';

Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldShowAlert: true,
    shouldPlaySound: true,
    shouldSetBadge: true,
  }),
});
```

### Registering for push notifications

The `registerForPushNotificationsAsync(jwtToken)` function should be called after the user logs in. It performs the following steps:

1. **Validates physical device** — uses `expo-device`'s `Device.isDevice` flag; simulators cannot receive Expo push tokens and will show an alert.
2. **Requests permissions** — calls `Notifications.getPermissionsAsync()` and, if permission has not already been granted, prompts the user with `Notifications.requestPermissionsAsync()`.
3. **Obtains the Expo push token** — calls `Notifications.getExpoPushTokenAsync()` to get the token string.
4. **Sends the token to the backend** — `POST /api/users/save-push-token` with the token in the request body and the JWT in the `Authorization` header.
5. **Configures an Android notification channel** (Android 8+ / API 26+) — creates a channel named `'default'` with `AndroidImportance.MAX`, a vibration pattern, and the DEC brand color (`#16a34a`).

```js
// Called after login
await registerForPushNotificationsAsync(jwtToken);
```

## Backend Storage

### The `Notification` model

Notifications are persisted in MongoDB using the `Notification` model defined in `backend/models/Notifications.js`:

```js
const NotificationSchema = new mongoose.Schema({
  title:       { type: String, required: true },
  body:        { type: String, required: true },
  pathologyId: { type: mongoose.Schema.Types.ObjectId, ref: 'Pathology' },
  location:    { type: String },
  imageUrl:    { type: String },
  sentAt:      { type: Date, default: Date.now },
  targetRoles: { type: [String], default: ['user', 'tecnico'] },
  active:      { type: Boolean, default: true },
});

NotificationSchema.index({ sentAt: -1 });
```

Key fields:

| Field | Description |
|---|---|
| `title` / `body` | The notification headline and message text. |
| `pathologyId` | Optional reference to the `Pathology` that triggered the alert. |
| `location` | Free-text location string (e.g. the outbreak area). |
| `targetRoles` | Array of roles that should receive this notification (`'user'`, `'tecnico'`, `'admin'`). |
| `active` | Set to `false` to hide outdated notifications without deleting them. |
| `sentAt` | Indexed in descending order for fast feed queries. |

### Syncing notifications to the device

When connectivity is restored, `syncService.js` calls `syncNotifications(token)`, which hits `GET /api/notifications/my` and returns the user's unread notifications as a JSON array. These can then be displayed in-app.

```js
// From syncService.js
export const syncNotifications = async (token) => {
  const response = await retryWithBackoff(async () => {
    return await api.get('notifications/my', {
      headers: { Authorization: `Bearer ${token}` }
    });
  }, 2);

  const notifications = Array.isArray(response.data)
    ? response.data
    : response.data.notifications || [];

  return notifications;
};
```

## Notification Flow

<Steps>
  <Step title="App requests notification permission">
    On first launch (after login), `registerForPushNotificationsAsync` checks the current permission status. If not yet granted, it prompts the user with the system permission dialog.
  </Step>
  <Step title="Expo SDK returns a push token">
    Once permission is granted on a physical device, `Notifications.getExpoPushTokenAsync()` returns a unique token of the form `ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]`.
  </Step>
  <Step title="Token is saved to the backend">
    The token is sent to `POST /api/users/save-push-token` with the user's JWT. The backend associates the token with the authenticated user's record in MongoDB.
  </Step>
  <Step title="Admin creates a notification">
    An admin user calls `POST /api/notifications/create` (requires `verifyToken` + `onlyAdmin` middleware) with a `title`, `body`, and optional `targetRoles`, `pathologyId`, `location`, and `expiresAt`. A new `Notification` document is saved to MongoDB.
  </Step>
  <Step title="Device receives and displays the notification">
    The Expo Push Service delivers the notification to registered devices. The foreground handler ensures the alert, sound, and badge are shown even if the app is open. Users can then open the app to view the full notification list.
  </Step>
</Steps>

## In-App Notification List

Users can view their notification feed inside the app. The `AlarmsScreen` (found under `src/screens/mobile/`) lists scheduled and received notifications, allowing users to review alerts related to disease outbreaks, treatment reminders, and admin broadcasts.

Notifications are fetched via `syncNotifications(token)` during the auto-sync cycle, so the list stays current whenever the device is online.

## Backend API Reference

All notification endpoints are mounted at `/api/notifications`.

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/notifications/create` | JWT + Admin | Creates a new notification targeted to one or more roles. |
| `GET` | `/api/notifications/my` | JWT | Returns unread notifications for the authenticated user's role, excluding expired ones. |
| `POST` | `/api/notifications/:id/read` | JWT | Marks a specific notification as read for the authenticated user by appending their `userId` to the `readBy` array. |

### `POST /api/notifications/create` — request body

```json
{
  "title": "Brote de Roya detectado",
  "body": "Se han registrado nuevos casos en la región norte.",
  "pathologyId": "664a1f2e3b4c5d6e7f8a9b0c",
  "location": "Zona Norte - Finca La Esperanza",
  "targetRoles": ["user", "tecnico"],
  "expiresAt": "2026-06-30T00:00:00.000Z"
}
```

### `GET /api/notifications/my` — response

```json
[
  {
    "_id": "664b2a3c4d5e6f7a8b9c0d1e",
    "title": "Brote de Roya detectado",
    "body": "Se han registrado nuevos casos en la región norte.",
    "location": "Zona Norte - Finca La Esperanza",
    "targetRoles": ["user", "tecnico"],

    "sentAt": "2026-05-01T10:00:00.000Z"
  }

```
