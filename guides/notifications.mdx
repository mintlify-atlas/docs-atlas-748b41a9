DEC includes a notification system that allows admins to broadcast alerts to users by role. Notifications are stored in MongoDB, fetched by the mobile client during the sync cycle, and displayed in-app. The groundwork for Expo push token registration is also present in `notificationService.js`, though it is not yet activated.

## Client-Side Setup with `notificationService.js`

The `notificationService.js` file configures how the app handles incoming notifications when the app is in the foreground.

### Foreground notification handler

The handler is configured to always display an alert, play a sound, and update the badge count when a notification arrives while the app is open:

```js
import * as Notifications from 'expo-notifications';

Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldShowAlert: true,
    shouldPlaySound: true,
    shouldSetBadge: true,
  }),
});
```

### Push token registration (scaffolded, not yet active)

`notificationService.js` contains a commented-out `registerForPushNotificationsAsync(jwtToken)` function that, when activated, would perform the following steps:

1. **Validates physical device** — uses `expo-device`'s `Device.isDevice` flag; simulators cannot receive Expo push tokens and will show an alert.
2. **Requests permissions** — calls `Notifications.getPermissionsAsync()` and, if permission has not already been granted, prompts the user with `Notifications.requestPermissionsAsync()`.
3. **Obtains the Expo push token** — calls `Notifications.getExpoPushTokenAsync()` to get the token string.
4. **Sends the token to the backend** — `POST /api/users/save-push-token` with the token in the request body and the JWT in the `Authorization` header.
5. **Configures an Android notification channel** (Android 8+ / API 26+) — creates a channel named `'default'` with `AndroidImportance.MAX`, a vibration pattern, and the DEC brand color (`#16a34a`).

<Note>
  The `registerForPushNotificationsAsync` function is currently **commented out** in `notificationService.js`. The DEC README marks push notifications as **"EN CONSTRUCCION"** (in progress). The foreground notification handler and the in-app notification feed (via `syncNotifications`) are functional, but end-to-end push delivery to device home screens is not yet wired up.
</Note>

## Backend Storage

### The `Notification` model

Notifications are persisted in MongoDB using the `Notification` model defined in `backend/models/Notifications.js`:

```js
const NotificationSchema = new mongoose.Schema({
  title:       { type: String, required: true },
  body:        { type: String, required: true },
  pathologyId: { type: mongoose.Schema.Types.ObjectId, ref: 'Pathology' },
  location:    { type: String },
  imageUrl:    { type: String },
  sentAt:      { type: Date, default: Date.now },
  targetRoles: { type: [String], default: ['user', 'tecnico'] },
  active:      { type: Boolean, default: true },
});

NotificationSchema.index({ sentAt: -1 });
```

Key fields:

| Field | Description |
|---|---|
| `title` / `body` | The notification headline and message text. |
| `pathologyId` | Optional reference to the `Pathology` that triggered the alert. |
| `location` | Free-text location string (e.g. the outbreak area). |
| `imageUrl` | Optional URL for an image related to the notification. |
| `targetRoles` | Array of roles that should receive this notification (`'user'`, `'tecnico'`). |
| `active` | Set to `false` to hide outdated notifications without deleting them. |
| `sentAt` | Indexed in descending order for fast feed queries. |

<Note>
  The `getUserNotifications` controller also queries by `expiresAt` and `readBy` in its MongoDB filter, but these fields are not declared in the `NotificationSchema`. They are treated as ad-hoc document fields — documents without them match as if `expiresAt` is `null` (never expired) and `readBy` is empty.
</Note>

### Syncing notifications to the device

When connectivity is restored, `syncService.js` calls `syncNotifications(token)`, which hits `GET /api/notifications/my` and returns the user's unread notifications as a JSON array. These are then available for display in-app.

```js
// From syncService.js
export const syncNotifications = async (token) => {
  const response = await retryWithBackoff(async () => {
    return await api.get('notifications/my', {
      headers: { Authorization: `Bearer ${token}` }
    });
  }, 2);

  const notifications = Array.isArray(response.data)
    ? response.data
    : response.data.notifications || [];

  return notifications;
};
```

## Notification Flow

<Steps>
  <Step title="Admin creates a notification">
    An admin user calls `POST /api/notifications/create` (requires `verifyToken` + `onlyAdmin` middleware) with a `title`, `body`, and optional `targetRoles`, `pathologyId`, `location`, and `expiresAt`. A new `Notification` document is saved to MongoDB.
  </Step>
  <Step title="Device reconnects and syncs">
    When the mobile app detects an internet connection, `startAutoSync` triggers `syncNotifications(token)`, which calls `GET /api/notifications/my`. The backend filters notifications by the user's role and excludes any that have expired or been marked read by that user.
  </Step>
  <Step title="Notifications are displayed in-app">
    The returned notification array is available to any screen that reads from the sync result. The foreground handler ensures alerts, sounds, and badge updates are shown when the app is open.
  </Step>
</Steps>

## Backend API Reference

All notification endpoints are mounted at `/api/notifications`.

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/notifications/create` | JWT + Admin | Creates a new notification targeted to one or more roles. |
| `GET` | `/api/notifications/my` | JWT | Returns unread, non-expired notifications for the authenticated user's role. |
| `POST` | `/api/notifications/:id/read` | JWT | Marks a specific notification as read for the authenticated user. |

### `POST /api/notifications/create` — request body

```json
{
  "title": "Brote de Roya detectado",
  "body": "Se han registrado nuevos casos en la región norte.",
  "pathologyId": "664a1f2e3b4c5d6e7f8a9b0c",
  "location": "Zona Norte - Finca La Esperanza",
  "targetRoles": ["user", "tecnico"],
  "expiresAt": "2026-06-30T00:00:00.000Z"
}
```

### `GET /api/notifications/my` — response

```json
[
  {
    "_id": "664b2a3c4d5e6f7a8b9c0d1e",
    "title": "Brote de Roya detectado",
    "body": "Se han registrado nuevos casos en la región norte.",
    "location": "Zona Norte - Finca La Esperanza",
    "targetRoles": ["user", "tecnico"],
    "active": true,
    "sentAt": "2026-05-01T10:00:00.000Z"
  }
]
```

<Warning>
  Expo push notifications require a **physical device** to generate a valid push token. `Notifications.getExpoPushTokenAsync()` will fail on iOS simulators and Android emulators. Always test notification registration on a real iOS or Android device.
</Warning>
